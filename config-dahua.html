1. Qué es la API CGI de Dahua y para qué la usa Limen

Dahua dispone de dos interfaces de programación para sus dispositivos de control de acceso:

APIDescripción¿La usa Limen?
CGI Interface
(/cgi-bin/)		 API HTTP/HTTPS embebida directamente en el firmware de cada dispositivo. Compatible con casi toda la gama de hardware Dahua con módulo de control de acceso. Autenticación HTTP Digest estándar (RFC 7616). No requiere software adicional — solo acceso de red al dispositivo. ✅ Sí
DHOP / DSHAPI API de nivel superior disponible en algunos modelos recientes con Android embebido (serie X y modelos premium). Más funcionalidades, pero no universal. Requiere firmware específico. ❌ No (no universal)
DSS / Smart PSS API API del software de gestión centralizada DSS Professional. Análoga a HikCentral — para instalaciones con servidor de gestión dedicado. ❌ No (requiere servidor DSS)
💡 Por qué CGI: La interfaz CGI de Dahua es el equivalente exacto de ISAPI en Hikvision — está embebida en el firmware, usa HTTP Digest, y es compatible con toda la gama de hardware AC. Limen usa la misma estrategia en ambos proveedores: sin software intermedio, directo al dispositivo.

Limen usa la CGI para las siguientes operaciones sobre el dispositivo Dahua:

  • Crear y actualizar personas (empleados, contratistas, visitas) con su identificador y nombre
  • Bloquear y desbloquear acceso de forma inmediata (finiquito, suspensión)
  • Asignar perfiles de acceso a puertas específicas
  • Abrir remotamente una puerta desde el dashboard de Limen
  • Leer eventos de acceso (entradas, salidas, rechazos) para el historial y auditoría
  • Listar las puertas/canales disponibles en el dispositivo
  • Subir foto facial (en terminales ASI con cámara de reconocimiento facial)
  • Registrar patentes vehiculares (en modelos con módulo LPR, no crítico)

Si devuelve información del sistema (modelo, firmware, número de serie), la CGI está activa y el dispositivo es compatible.

3. Prerequisitos

  1. Firmware compatible. La interfaz CGI de control de acceso requiere firmware ≥ V2.600.X en la mayoría de modelos. Verificar en la interfaz web del dispositivo: Sistema → Información del sistema → Versión de software.
  2. CGI habilitado. En dispositivos recientes está habilitado por defecto. Si no responde, verificar en Configuración → Red → Servicio → CGI.
  3. Acceso de red desde el servidor Limen al dispositivo. El servidor donde corre Limen debe poder alcanzar la IP del dispositivo en el puerto 80 (HTTP) o 443 (HTTPS). Si el dispositivo está en otra VLAN, verificar ACLs de red.
  4. Usuario con permisos de Access Control Management y Control Remoto. No es necesario usar admin; se puede crear un usuario dedicado limen con solo los permisos necesarios. En la interfaz web del dispositivo: Sistema → Cuenta → Agregar usuario. Marcar los grupos: Control de acceso y Control remoto.
  5. HTTPS recomendado en producción. Si el dispositivo usa HTTPS con certificado auto-firmado, Limen lo acepta (igual que Hikvision). Usar la URL con https://.

4. Habilitar la API CGI en el dispositivo

En la mayoría de dispositivos Dahua la CGI está habilitada por defecto. Para verificar y activar:

Vía interfaz web del dispositivo

  1. Acceder a http://<IP-del-dispositivo> en el navegador
  2. Ir a Configuración → Red → Conexión (o Servicio según el firmware)
  3. Verificar que CGI o Web Service está habilitado
  4. Guardar y reiniciar si se modificó algún parámetro

Vía Smart PSS (software de gestión local)

  1. Seleccionar el dispositivo → Configuración remota → Red → Servicios
  2. Verificar que CGI está marcado como activo
💡 Verificar con curl antes de configurar en Limen: 
curl --digest -u admin:Contraseña \
  "http://192.168.1.108/cgi-bin/magicBox.cgi?action=getSystemInfo"
Respuesta esperada: líneas key=value con modelo, firmware y número de serie. Si responde, la API CGI está activa y las credenciales son correctas.

5. Configurar en Limen

  1. Ir a Administración → Integraciones en Limen
  2. En la tarjeta Control de Acceso Físico, seleccionar proveedor Dahua Access Control
  3. Completar los campos:
    CampoDescripciónEjemplo
    URL Base IP o hostname del dispositivo Dahua. Si usa HTTPS con certificado auto-firmado, usa https:// (Limen acepta certificados auto-firmados). Sin barra final. http://192.168.1.108
    https://dahua.empresa.local
    Usuario Usuario del dispositivo con permisos de Access Control Management y Control Remoto. Se recomienda crear un usuario dedicado limen en lugar de usar admin. admin o limen
    Contraseña Contraseña del usuario del dispositivo. Se cifra con DataProtection antes de persistir en la base de datos. Nunca se muestra nuevamente después de guardar. Admin1234!
  4. Hacer clic en Probar conexión para verificar que las credenciales son válidas
  5. Hacer clic en Guardar
💡 Diferencia con Hikvision: Los campos son idénticos (URL Base, Usuario, Contraseña) porque ambos usan HTTP Digest. La diferencia es interna: Limen llama a endpoints /cgi-bin/ en lugar de /ISAPI/. Para el administrador, la configuración es la misma.

6. Endpoints CGI implementados

Esta es la tabla exacta de qué endpoint CGI usa cada método de Limen:

Método LimenMétodo HTTPEndpoint CGINotas
SyncPersonAsync POST /cgi-bin/AccessControl.cgi?action=insertRecordList&name=UserInfoDetail
Fallback: /cgi-bin/AccessControl.cgi?action=updateRecordList&name=UserInfoDetail 		Crea la persona con insertRecordList. Si el userID ya existe (error de duplicado), reintenta con updateRecordList. Si se provee foto, se hace segunda llamada a FaceInfoManager.cgi (no crítico).
BlockUserAsync POST /cgi-bin/AccessControl.cgi?action=updateRecordList&name=UserInfoDetail Establece validEnd = "2000.01.01 00:00:00" (fecha pasada → no puede ingresar). Fallback: establece userType = "2" (lista negra en algunos modelos) si el primer intento falla.
UnblockUserAsync POST /cgi-bin/AccessControl.cgi?action=updateRecordList&name=UserInfoDetail Restaura validEnd = "2030.12.31 23:59:59" y userType = "0" (normal).
AssignAccessProfileAsync POST /cgi-bin/AccessControl.cgi?action=updateRecordList&name=UserInfoDetail El profileName se interpreta como ID de puerta (campo doors). Si no es numérico, se usa 0 (puerta por defecto). Las puertas deben estar configuradas en el dispositivo.
RemoteOpenDoorAsync POST (+ GET como fallback) /cgi-bin/AccessControl.cgi?action=openDoor&channel={n} El doorHardwareId debe ser el número de canal (entero ≥ 1). Algunos modelos responden solo a GET para este endpoint — Limen intenta POST primero y si falla HTTP-status, reintenta con GET.
RegisterVehiclePlateAsync POST /cgi-bin/LPR.cgi?action=add Solo disponible en terminales con módulo LPR (ej: ITC237 series). En controladoras de acceso puro (DHI-ASC) no existe este endpoint. Si falla, Limen registra warning y retorna éxito parcial — no bloquea el flujo principal.
GetAccessEventsAsync POST /cgi-bin/RecordFinder.cgi?action=find&name=AccessControlCardRec Paginado interno con campo Position (100 eventos por página). Máximo 2000 eventos por llamada. Filtra por StartTime/EndTime en formato yyyy-MM-dd HH:mm:ss (hora local del dispositivo). Campos de evento: UserID, UserName, CardNo, DoorID, Time, Method, Type (0=entrada, 1=salida), Status (0=concedido, ≠0=denegado).
GetHardwareDoorsAsync GET /cgi-bin/AccessControl.cgi?action=getConfig&name=DoorConfig (intento 1)
/cgi-bin/AccessControl.cgi?action=getConfig&name=DoorConfig&channel={n} n=1..8 (fallback) 		Primero intenta la lista completa. Si no está disponible, itera los canales 1 a 8 (máximo típico en controladoras Dahua). La respuesta puede estar en formato JSON o key=value propietario — Limen maneja ambos formatos.
SyncPersonAsync (foto facial) POST /cgi-bin/FaceInfoManager.cgi?action=add Solo en terminales ASI con cámara de reconocimiento facial. En controladoras puras (DHI-ASC) no existe este endpoint. Si falla, Limen registra warning y continúa — la persona sigue creada.

7. Mapeo de canales y puertas (HardwareId)

En Limen, cada puerta tiene un campo Hardware ID que identifica la puerta en el sistema de control de acceso físico. Para Dahua CGI, ese ID debe ser el número de canal entero (1, 2, 3...) como está configurado en el dispositivo.

Numeración de canales en Dahua

A diferencia de algunas APIs que usan índice 0 (base 0), la CGI de Dahua usa canales con base 1 para la apertura remota (openDoor?channel=1). En otros endpoints como los de configuración de puertas puede verse como índice 0 internamente, pero Limen normaliza todo a base 1 para consistencia con los demás proveedores.

ModeloCanales disponiblesHardwareId en Limen
DHI-ASC1204C / DHI-ASC2204C (4 puertas) Canal 1, 2, 3, 4 "1", "2", "3", "4"
DHI-ASC2202B (2 puertas) Canal 1, 2 "1", "2"
Terminal ASI (1 puerta) Canal 1 "1"

¿Cómo saber el número de cada puerta?

  1. En Limen: ir a Administración → Puertas → Importar desde hardware. Limen consulta el dispositivo con GetHardwareDoorsAsync y lista los canales disponibles para que los mapees a las zonas de acceso.
  2. Alternativamente, verificar directamente con curl:
    curl --digest -u admin:Contraseña \
      "http://192.168.1.108/cgi-bin/AccessControl.cgi?action=getConfig&name=DoorConfig&channel=1"
  3. En la interfaz web del dispositivo: Control de Acceso → Configuración de puertas muestra el listado con el número de cada puerta.
⚠️ Importante: El doorHardwareId en Limen debe ser un número entero ≥ 1. Si envías un UUID o un string no numérico, RemoteOpenDoorAsync falla con un mensaje claro antes de hacer ninguna request al dispositivo.

8. Autenticación HTTP Digest

La CGI de Dahua usa HTTP Digest Authentication estándar (RFC 7616), exactamente igual que Hikvision ISAPI. A diferencia de Basic Auth, las credenciales nunca viajan en texto plano: el dispositivo responde con un reto (nonce) y el cliente responde con un hash MD5/SHA-256 de las credenciales + el nonce.

En .NET, HttpClientHandler maneja esto automáticamente cuando se le configuran Credentials = new NetworkCredential(user, pass). Limen establece además PreAuthenticate = true para evitar el round-trip inicial en llamadas subsecuentes (mejora de latencia en operaciones consecutivas).

Certificados auto-firmados

Los dispositivos Dahua en instalaciones on-premise generalmente usan certificados auto-firmados cuando se configura HTTPS. Limen acepta cualquier certificado de dispositivo (DangerousAcceptAnyServerCertificateValidator), lo que es correcto en redes internas controladas.

En producción con requerimientos de seguridad estrictos, lo recomendado es: instalar un certificado firmado por el CA corporativo en el dispositivo, o usar HTTP en una VLAN aislada sin acceso externo.

9. Troubleshooting

SíntomaCausa probableSolución
HTTP 401 Unauthorized Credenciales incorrectas o usuario sin permisos en el dispositivo Verificar usuario/contraseña con curl: curl --digest -u admin:contraseña http://IP/cgi-bin/magicBox.cgi?action=getSystemInfo. Confirmar que el usuario tiene grupos "Control de Acceso" y "Control Remoto" en Sistema → Cuenta del dispositivo.
Connection refused / timeout Puerto bloqueado por firewall o CGI deshabilitado Verificar conectividad: telnet 192.168.1.108 80. Verificar que CGI está habilitado en Configuración → Red → Servicio del dispositivo.
Respuesta Error. Code=8 El userID ya existe al intentar insertRecordList Comportamiento esperado. Limen reintenta automáticamente con updateRecordList. Si el error persiste en el update, verificar que el formato del userID es correcto (algunos modelos limitan a 8 caracteres numéricos).
Respuesta Error. Code=4 Error de parámetro — el payload JSON tiene un campo no reconocido Revisar logs de Limen para ver el payload exacto. Puede ocurrir si el firmware es muy antiguo. Actualizar firmware del dispositivo o reportar al soporte de Limen para ajustar el payload.
RemoteOpen retorna OK pero la puerta no abre El número de canal es incorrecto, o el relé no está conectado Verificar que el doorHardwareId en Limen coincide con el canal físico del dispositivo. Probar directamente con curl (ver sección de verificación). Verificar el estado del relé en la interfaz web del dispositivo.
GetHardwareDoors retorna lista vacía El endpoint de lista no responde o el formato es desconocido Limen intenta automáticamente la iteración por canal 1..8. Si la lista sigue vacía, agregar las puertas manualmente en Limen → Administración → Puertas con el número de canal correcto.
Foto facial no se sube El dispositivo no tiene módulo de reconocimiento facial o no soporta FaceInfoManager.cgi Comportamiento esperado en controladoras puras (DHI-ASC). Solo terminales ASI series con cámara soportan este endpoint. El acceso por tarjeta/PIN sigue funcionando normalmente.
LPR.cgi retorna HTTP 404 El dispositivo no tiene módulo LPR Comportamiento esperado en controladoras y terminales sin módulo de reconocimiento de patentes. Limen registra warning en logs y continúa. El flujo de registro de personas no se afecta.
Eventos de acceso traen timestamps incorrectos El dispositivo está en zona horaria diferente al servidor Limen La CGI de Dahua usa hora local del dispositivo en el campo Time. Verificar que el dispositivo tiene la zona horaria correcta: Sistema → Configuración general → Hora del sistema. Considerar sincronizar el dispositivo con un servidor NTP.

10. Diferencias con Hikvision ISAPI

Dahua CGI y Hikvision ISAPI son similares en concepto (API embebida, HTTP Digest) pero difieren en los detalles de los endpoints y formatos. Esta tabla resume las diferencias clave:

AspectoHikvision ISAPIDahua CGI
Prefijo de URLs /ISAPI/AccessControl/ /cgi-bin/AccessControl.cgi?action=...
Formato de respuesta Siempre JSON (con ?format=json) o XML JSON o texto key=value propietario según el endpoint
Indicador de éxito {"statusCode": 1, "statusString": "OK"} Texto "OK" o JSON {"result": true} o HTTP 200 sin body
Crear persona POST /ISAPI/AccessControl/UserInfo/Record?format=json POST /cgi-bin/AccessControl.cgi?action=insertRecordList&name=UserInfoDetail
Bloquear usuario Campo Valid.enable = false via SetUp Campo validEnd con fecha pasada (2000.01.01) via updateRecordList
Apertura remota PUT /ISAPI/AccessControl/RemoteControl/door/{n} con body XML POST /cgi-bin/AccessControl.cgi?action=openDoor&channel={n} sin body
Foto facial /ISAPI/Intelligent/FDLib/FaceDataRecord /cgi-bin/FaceInfoManager.cgi?action=add
Consulta de eventos POST /ISAPI/AccessControl/AcsEvent?format=json
Timestamps: ISO 8601 UTC 		POST /cgi-bin/RecordFinder.cgi?action=find&name=AccessControlCardRec
Timestamps: yyyy-MM-dd HH:mm:ss (hora local del dispositivo)
Listado de puertas GET /ISAPI/AccessControl/Door/param?format=json GET /cgi-bin/AccessControl.cgi?action=getConfig&name=DoorConfig
Capacidades del dispositivo GET /ISAPI/AccessControl/capabilities?format=json para saber max puertas No hay endpoint estándar — iteración fija 1..8 como fallback
Autenticación HTTP Digest RFC 7616 — mismo mecanismo HTTP Digest RFC 7616 — mismo mecanismo
Certificados SSL Auto-firmados aceptados (configurable) Auto-firmados aceptados (configurable)

Verificación manual con curl

Verificar conectividad y credenciales

curl --digest -u admin:Contraseña \
  "http://192.168.1.108/cgi-bin/magicBox.cgi?action=getSystemInfo"

Crear un usuario de prueba

curl --digest -u admin:Contraseña -X POST \
  -H "Content-Type: application/json" \
  -d '{"UserInfoDetail":{"userInfo":[{"userID":"9999","userName":"Test Limen","userType":"0","userVerifyMode":"","validEnd":"2030.12.31 23:59:59","doors":[0]}]}}' \
  "http://192.168.1.108/cgi-bin/AccessControl.cgi?action=insertRecordList&name=UserInfoDetail"

Respuesta esperada: OK o {"result":true}

Abrir canal 1 remotamente

curl --digest -u admin:Contraseña -X POST \
  "http://192.168.1.108/cgi-bin/AccessControl.cgi?action=openDoor&channel=1"

Leer eventos de acceso del último día

curl --digest -u admin:Contraseña -X POST \
  -H "Content-Type: application/json" \
  -d '{"Condition":{"StartTime":"2026-05-17 00:00:00","EndTime":"2026-05-18 23:59:59","Dirs":[0],"Count":20,"Position":0}}' \
  "http://192.168.1.108/cgi-bin/RecordFinder.cgi?action=find&name=AccessControlCardRec"

Listar puertas configuradas

curl --digest -u admin:Contraseña \
  "http://192.168.1.108/cgi-bin/AccessControl.cgi?action=getConfig&name=DoorConfig"
