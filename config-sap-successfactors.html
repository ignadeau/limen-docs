1. Qué es SAP SuccessFactors Employee Central

SAP SuccessFactors Employee Central es el módulo de Recursos Humanos de SAP que gestiona la nómina de empleados, contratos, altas, bajas y estructura organizacional. Es el sistema de registro (System of Record) de RRHH para organizaciones que utilizan el ecosistema SAP a nivel corporativo.

La integración de Limen con SuccessFactors permite:

  • Mantener sincronizada la nómina activa de empleados con los perfiles de acceso físico.
  • Detectar automáticamente bajas laborales y bloquear las credenciales de acceso en el sistema de control físico.
  • Validar en tiempo real si un RUT corresponde a un empleado activo antes de autorizar acceso.
MecanismoCuándo actúaLatencia típica
Polling periódico Sincronización completa de nómina activa (job cada 15 min) ≤ 15 min
Validación en acceso Verificación de empleado activo al presentar credencial < 2 seg
Diferencia con GeoVictoria y Buk: SuccessFactors no ofrece webhooks salientes para eventos de baja. La sincronización es exclusivamente por polling. Si necesitas latencia inferior a 15 minutos para bloqueos de baja, considera complementar con un workflow en SuccessFactors que notifique a Limen mediante el módulo de Integration Center (configuración adicional fuera del alcance de esta guía).

2. Prerequisitos

2.1 Versión de SuccessFactors

Esta integración utiliza OData v2 API de Employee Central. Requiere:

  • SuccessFactors Employee Central habilitado (no aplica solo con módulos de Performance o Learning).
  • API OData habilitada para el Company ID (activado por defecto en instancias modernas).
  • Acceso de administración para crear OAuth2 Client Applications.

2.2 Permisos necesarios en SuccessFactors

El usuario administrador que realice la configuración necesita los siguientes permisos en Admin Center → Permission Settings:

PermisoUbicación en Admin Center
Manage OAuth2 Client Applications Admin Center → Manage Security → OAuth2 Client Application
Employee Central – Read access a PerPerson Admin Center → Manage Permission Roles → Data Access → Employee Central
Ver campos custom de persona (custom01..custom15) Mismo rol, sección "National ID Information" o "Personal Information"

3. Crear el OAuth2 Client Application en SuccessFactors

Limen usa OAuth 2.0 Client Credentials (server-to-server). No requiere que un usuario humano inicie sesión; el sistema se autentica directamente con un Client ID y Client Secret.

3.1 Acceder a OAuth2 Client Applications

  1. Ingresar a SuccessFactors como administrador.
  2. Ir a Admin Center (ícono de llave inglesa o engranaje en la barra superior).
  3. Buscar en el buscador: "OAuth".
  4. Seleccionar Manage OAuth2 Client Applications.

3.2 Crear la aplicación cliente

  1. Hacer clic en Register Client Application.
  2. Completar el formulario:
CampoValor recomendado
Application Name Limen - Control de Acceso
Description Integración con plataforma Limen para sincronización de nómina y control de acceso físico
Application URL URL de tu instancia Limen (ej: https://tuempresa.limen.app)
Bind to Users No (server-to-server, sin usuario asociado)
Token Validity (seconds) 1800 (30 minutos; Limen cachea el token y lo renueva automáticamente)
  1. Hacer clic en Generate para obtener el Client Secret (o generar un certificado X.509 si tu política de seguridad lo requiere).
  2. Copiar y guardar el Client Secret de inmediato. Solo se muestra una vez. Si lo pierdes, deberás regenerarlo.
  3. Anotar el Client ID generado (identificador alfanumérico que aparece junto al nombre de la aplicación).
Seguridad: Crear un Client Application dedicado exclusivamente para Limen. No reutilizar credenciales de otras integraciones (SAP Integration Center, SAP CPI, etc.). Si las credenciales se comprometen, se pueden revocar sin afectar otras integraciones.

3.3 Obtener el Company ID

  1. En Admin Center, buscar "Company Information".
  2. Seleccionar Basic Company Information.
  3. El campo Company ID es el identificador único de tu instancia (ej: MIEMPRESA_CHILE o un código numérico).
  4. Anotar este valor — se necesita para el flujo OAuth2.

4. Encontrar el Campo que Almacena el RUT

SuccessFactors no tiene un campo estándar llamado "RUT". El identificador tributario chileno se almacena en un campo custom del objeto PerPerson. El nombre exacto varía según la configuración de cada implementación de SuccessFactors.

4.1 Opciones comunes

Campo ODataDescripciónProbabilidad
custom01 Primer campo custom de PerPerson. Muy usado para documentos nacionales. Alta
custom02 Segundo campo custom (si custom01 está ocupado por otro dato). Media
personIdExternal ID externo del empleado. Algunas implementaciones lo populan con el RUT. Media
nationalId Disponible en el objeto PerNationalId (entity separada, requiere expand adicional). Baja (estructura diferente)

4.2 Cómo verificar el campo correcto

  1. Ejecutar la siguiente query OData en el navegador (con tu Company ID y credenciales):
    GET https://api10.successfactors.com/odata/v2/PerPerson?$top=1&$format=json
    (Reemplazar api10 por el datacenter de tu instancia.)
  2. Revisar la respuesta JSON: buscar el campo que contiene el RUT en formato 12345678-9, 12.345.678-9 o similar.
  3. El nombre de ese campo es el valor a ingresar en Campo RUT (OData) en Limen.
Tip: Si no tienes acceso directo a la API OData, consulta al equipo funcional de SuccessFactors de tu empresa. Ellos tienen acceso a la herramienta Metadata Framework (MDF) y pueden identificar en qué campo custom se almacena el RUT.

5. Configurar la Integración en Limen

  1. Ingresar a Limen como TenantAdmin.
  2. Ir a Administración → Integraciones → RRHH → SAP SuccessFactors.
  3. Completar los campos:
CampoDescripciónEjemplo
URL de la API URL base del datacenter donde está alojada tu instancia SF. Sin barra final. https://api10.successfactors.com
Client ID Identificador del OAuth2 Client Application creado en el paso 3. limen-prod-client
Client Secret Secret del OAuth2 Client Application. Se cifra al guardar; no se muestra después. eyJhbGciOi...
Company ID Identificador de la empresa en SuccessFactors (obtenido en el paso 3.3). MIEMPRESA_CHILE
Campo RUT (OData) Nombre del campo de PerPerson que almacena el RUT (obtenido en el paso 4). custom01
  1. Hacer clic en Guardar.
  2. Hacer clic en Probar Conexión. Debe mostrar: "Conexión verificada correctamente. N empleados activos."
Seguridad: Las credenciales se cifran con AES-256 (Microsoft DataProtection) antes de almacenarse en la base de datos. El Client Secret nunca queda en texto plano ni en logs ni en appsettings.

6. Endpoints OData Implementados

Limen consume los siguientes endpoints de la API OData v2 de SuccessFactors:

OperaciónEndpoint ODataParámetros clave
Obtener token OAuth POST /oauth/token grant_type=client_credentials, company_id
Empleados activos GET /odata/v2/PerPerson $select=personIdExternal,{rutField}
$expand=employmentNav
$top=200&$skip=N
$format=json
Test de conexión GET /odata/v2/PerPerson $top=1&$format=json

El filtro de empleados activos vs. desvinculados se aplica en Limen evaluando el campo endDate del employmentNav:

  • endDate nulo o ausente → empleo vigente → empleado activo.
  • endDate con fecha futura → empleo vigente → empleado activo.
  • endDate con fecha pasada → empleo terminado → empleado desvinculado.

7. Cómo Funciona la Sincronización

7.1 Flujo de sincronización periódica

  1. Cada 15 minutos, el job IdentitySyncJob de Limen invoca IHrSystemClient.GetActiveEmployeesAsync().
  2. Limen verifica el token OAuth en cache; si expiró, lo renueva automáticamente (POST a /oauth/token).
  3. Se pagina PerPerson de 200 en 200 registros hasta obtener toda la nómina.
  4. Los RUT obtenidos se comparan con los empleados activos en Limen:
    • RUT nuevo en SF, no en Limen: se crea el perfil y se asigna acceso en la plataforma física (ZK/Hikvision).
    • RUT en Limen pero no en SF: se bloquea el acceso físico y se marca como desvinculado.
    • RUT en ambos: sin acción (a menos que haya cambios en nombre/departamento).

7.2 Normalización del RUT

SAP SuccessFactors puede almacenar el RUT en distintos formatos según la configuración: "12.345.678-9", "12345678-9" o "123456789". Limen normaliza todos los formatos a sin puntos ni guión antes de comparar, por lo que la integración funciona independientemente del formato elegido.

Formato en SFNormalizado por Limen
12.345.678-9123456789
12345678-9123456789
12345678-K12345678K
123456789123456789

7.3 Caché del token OAuth

El token se cachea en memoria por expires_in - 60 segundos (típicamente 1740 s ≈ 29 minutos). Un minuto antes de que expire, la siguiente request obtiene un nuevo token automáticamente. No hay rotación activa de tokens; si el servidor de SuccessFactors no está disponible al momento de renovar, la operación retorna lista vacía y reintenta en el próximo ciclo.

8. Troubleshooting

ErrorCausa probableSolución
HTTP 401 Unauthorized al obtener token Client ID, Client Secret o Company ID incorrectos. Verificar los tres valores. El Company ID distingue mayúsculas.
HTTP 401 al llamar a PerPerson (ya con token) Token expirado o el Client App no tiene permisos de lectura en Employee Central. Limen limpia el cache y reintenta una vez. Si persiste, revisar los permisos del role asignado al Client App en Admin Center.
HTTP 403 Forbidden El Client App no tiene acceso al objeto PerPerson o al campo de RUT configurado. En Admin Center → Manage Permission Roles, asegurarse de que el rol asociado al Client App tiene acceso de lectura a Personal Information y al campo custom específico.
HTTP 429 Too Many Requests Se excedió el rate limit de la API SuccessFactors. Limen loguea el evento y retorna los registros parciales acumulados. La sincronización completa se completará en el próximo ciclo de 15 minutos. Considera aumentar el intervalo de polling si la nómina es muy grande.
Campo RUT vacío en todos los empleados El valor de Campo RUT (OData) configurado en Limen no coincide con el campo real en SuccessFactors. Revisar el paso 4 de esta guía. Los logs de Limen mostrarán "[SAP-SF] Campo 'custom01' no encontrado en PerPerson" si el nombre es incorrecto.
Timeout en la prueba de conexión El datacenter de SuccessFactors no es accesible desde el servidor Limen. Verificar que el servidor Limen tiene salida a internet hacia api10.successfactors.com (o el datacenter correspondiente) en el puerto 443. Revisar reglas de firewall y proxy corporativo.
Nómina parcial: menos empleados de lo esperado Limitación de paginación o campos de empleos sin endDate que SuccessFactors interpreta como terminados. Revisar en SF que los contratos indefinidos tienen endDate nulo (no con fecha futura muy lejana, que podría interpretarse como terminado si hay un bug en la data).
Error invalid_client en el token El Client App fue revocado o el secret fue regenerado en SF. Ir a Admin Center → Manage OAuth2 Client Applications → regenerar el secret → actualizar las credenciales en Limen.

8.1 Dónde ver los logs

Todos los mensajes del cliente SAP se prefijan con [SAP-SF]. Para filtrar en los logs de Limen:

# PowerShell — ver logs en tiempo real filtrados por SAP-SF:
Get-Content -Path "C:\Limen\logs\limen-.log" -Wait | Select-String "\[SAP-SF\]"

9. Comparativa con Buk y GeoVictoria

Característica SAP SuccessFactors Buk GeoVictoria
Autenticación OAuth 2.0 Client Credentials API Key estático Login con ConsumerKey/Secret → Bearer token
Protocolo de datos OData v2 (REST/JSON) REST/JSON REST/JSON
Webhooks/eventos en tiempo real No (solo polling) Sí (baja laboral < 30 seg) No (solo polling)
Latencia de sincronización ≤ 15 minutos < 30 segundos para bajas ≤ 15 minutos
Campo RUT Configurable (custom01..custom15 o personIdExternal) Campo rut estándar Campo identifier
Perfil de cliente Empresas con ecosistema SAP corporativo (minería, banca, industria pesada) PyMEs y medianas empresas chilenas Empresas con asistencia/turnos en Chile
Paginación $top/$skip (OData estándar), 200 por página page/page_size, 100 por página Sin paginación (retorna todo en una llamada)
Necesidad de configuración adicional Alta (crear OAuth App, mapear campo RUT) Media (API Key + webhook) Baja (URL + credenciales)
Recomendación: Usa SAP SuccessFactors cuando la empresa ya tiene SAP como sistema de RRHH. Si el cliente usa Buk o GeoVictoria como RRHH, prefiere esas integraciones por ser más simples y con soporte de webhooks para bajas inmediatas.
