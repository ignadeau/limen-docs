Sincronización en Tiempo Real y Bloqueos de Seguridad Críticos

1. Arquitectura del Webhook

A diferencia de la sincronización programada (polling), los Webhooks permiten que GeoVictoria "empuje" la información a Limen en el instante exacto en que ocurre el evento. Esto es vital para las Bajas Laborales.

2. Configuración del Endpoint en Limen

Usted debe configurar en el panel de GeoVictoria la URL de destino para los eventos. Limen expone un endpoint dedicado:

https://api.limen.app/api/webhooks/geovictoria

3. Eventos Soportados y Payloads

Limen procesa principalmente el evento de "Cese de Funciones" o "Término de Contrato". El formato esperado es JSON:

{
  "EventType": "EMPLOYEE_TERMINATED",
  "Timestamp": "2026-05-05T14:30:05Z",
  "Data": {
    "Identifier": "12345678-9",
    "FullName": "Juan Perez",
    "Reason": "Termino de Contrato",
    "EffectiveDate": "2026-05-05T15:00:00Z"
  }
}

4. Seguridad y Verificación (Firma HMAC)

Para evitar que terceros envíen falsas bajas, Limen valida la firma del Webhook:

  • GeoVictoria envía un header X-GV-Signature.
  • Limen recalcula el hash usando el Webhook Secret configurado.
  • Si no coinciden, la petición se descarta y se registra como intento de intrusión en el AuditLog.

5. La Lógica de Bloqueo Inmediato (< 30s)

Una vez validado el Webhook, Limen ejecuta el siguiente flujo orquestado:

Etapa Acción Técnica Tiempo Est.
Mapeo Busca al usuario por RUT en ApplicationDbContext. 100ms
Inhabilitación Marca al usuario como IsActive = false en Limen. 50ms
Hardware Sync Envía comando de bloqueo a ZK/Hikvision vía IAccessControlClient. 2-5s
Revocación Digital Invalida tokens JWT y Credencial Móvil en Mobile.Api. 500ms

6. Monitoreo y Reintentos

Si el comando de bloqueo al hardware falla (ej: controladora offline), Limen:

  1. Mueve el evento a una Dead Letter Queue (DLQ).
  2. Reintenta cada 60 segundos hasta confirmar el éxito.
  3. Alerta al equipo de seguridad mediante un aviso de alta prioridad en el Dashboard: "Baja fallida en Hardware: [Nombre_Usuario]".
Prueba de Integración: Puede simular este evento usando Postman en el ambiente de Staging. Asegúrese de incluir la firma correcta en los headers para que el middleware de seguridad de Limen permita el paso.

7. Análisis de Logs de una Baja Exitosa

A continuación se presenta la secuencia de logs que debe observar en Serilog o la consola de Limen para confirmar que la "Regla de los 30 Segundos" se cumplió satisfactoriamente:

[14:30:05 INF] [GeoVictoriaWebhook] Received EMPLOYEE_TERMINATED event for RUT: 12345678-9
[14:30:05 DBG] [Security] Signature verification SUCCESS for Tenant: CLIENTE_PROD
[14:30:05 INF] [IdentityService] User identified: 'Juan Perez' (ID: d4e5f6...). Starting revocation flow.
[14:30:05 INF] [Domain] Applying status: IsActive = FALSE for Person: 12345678-9
[14:30:06 INF] [HardwareControl] Dispatching BLOCK_USER command to ZK_SERVER_01 (DoorIDs: 1, 2, 3)
[14:30:08 INF] [HardwareControl] ZK_SERVER_01 CONFIRMED: User 12345678-9 blocked successfully.
[14:30:08 INF] [MobileApi] Revoking JWT Tokens and Mobile Credentials for User: Juan Perez
[14:30:09 INF] [Audit] SEC_REVOCATION_COMPLETE: Access fully revoked for 12345678-9 in 4.2 seconds.

Interpretación de Estados Críticos

  • [Security] Signature verification: Si aparece como FAILED, verifique que el Shared Secret en Limen coincida con el configurado en GeoVictoria.
  • [HardwareControl] CONFIRMED: Es el log más importante. Indica que el hardware físico ya tiene la instrucción de bloqueo y la persona no podrá pasar aunque use su tarjeta física.
  • SEC_REVOCATION_COMPLETE: Este log de auditoría cierra el caso y es el que se utiliza para reportes de cumplimiento legal y KPIs de seguridad.
