BUK INTEGRATION

Sincronización en Tiempo Real — Bajas Laborales en menos de 30 segundos

← Volver al Índice Técnico

1. Arquitectura de Integración

La integración con Buk combina dos mecanismos complementarios para garantizar sincronización completa y reacción inmediata ante bajas laborales:

MecanismoCuándo actúaLatencia
Polling periódico Sincronización completa de nómina activa (job cada 15 min) ≤ 15 min
Webhooks en tiempo real Evento de baja laboral, alta o actualización de datos < 30 seg
Ventaja frente a GeoVictoria: Buk notifica activamente a Limen cuando ocurre una baja. No hay que esperar el próximo ciclo de polling — el bloqueo físico en los torniquetes se ejecuta en segundos desde el finiquito.

2. Obtener Credenciales en el Panel de Buk

2.1 API Key (auth_token)

El API Key es el token de autenticación para las consultas salientes de Limen hacia Buk.

  1. Ingresar al panel de Buk como administrador: https://[su-empresa].buk.cl
  2. Ir a Configuración → Integraciones → Acceso a la API
  3. Hacer clic en Generar nuevo token
  4. Asignar un nombre descriptivo: "Limen - Control de Acceso"
  5. Seleccionar el scope mínimo necesario:
    • employees:read — lectura de nómina
  6. Copiar el token generado. Solo se muestra una vez.
Buenas prácticas de seguridad: Generar un token con scope limitado exclusivamente a lectura de empleados. No reutilizar el token de otras integraciones. Si el token se expone accidentalmente, revocarlo desde el mismo panel.

2.2 Webhook Secret (Bearer Token para webhooks entrantes)

Buk autentica sus webhooks salientes usando un Bearer Token estático que usted configura. Limen verifica este token en cada webhook recibido para garantizar que proviene de Buk y no de un tercero.

  1. Generar un token secreto seguro. Puede usar:
    # PowerShell:
    [Convert]::ToBase64String([Security.Cryptography.RandomNumberGenerator]::GetBytes(32))

    # Linux/Mac:
    openssl rand -base64 32
  2. Guardar este valor — lo necesitará tanto en Limen como en el panel de Buk.

3. Configurar la Integración en Limen

  1. Ingresar a Limen como TenantAdmin.
  2. Ir a Administración → Integraciones → RRHH → Buk.
  3. Completar los campos:
CampoValorEjemplo
Tenant Name Subdominio de su cuenta Buk (sin .buk.cl) miempresa
API Key Token auth_token generado en el paso 2.1 eyJhbGciOi...
Webhook Secret Bearer Token secreto generado en el paso 2.2 aB3dE5fG7h...
País País de la instancia Buk (afecta la URL del endpoint) chile
  1. Hacer clic en Guardar.
  2. Hacer clic en Probar Conexión. Debe mostrar: "Conexión exitosa. N empleados activos."
Seguridad: Las credenciales se cifran con AES-256 (DataProtection) antes de almacenarse en la base de datos. No se guardan en texto plano ni en appsettings.

4. Configurar Webhooks en el Panel de Buk

Este paso configura a Buk para que notifique a Limen cada vez que ocurra un evento laboral relevante (baja, alta, actualización de datos).

4.1 Acceder a la configuración de webhooks

  1. En el panel de Buk, ir a Configuración → Integraciones → Webhooks
  2. Hacer clic en Nuevo Webhook

4.2 Configurar la URL de destino

CampoValor
URL del Webhook https://api.limen.app/api/webhooks/buk
Método HTTP POST
Formato del Payload application/json

4.3 Configurar la autenticación Bearer Token

En la sección de Autenticación del webhook:

  1. Seleccionar tipo: Bearer Token
  2. Ingresar el mismo valor del Webhook Secret configurado en Limen (paso 3)
  3. Buk enviará automáticamente Authorization: Bearer {token} en cada request

4.4 Configurar el header X-Tenant-Id (obligatorio)

Limen es multi-tenant. Para identificar a qué empresa corresponde el webhook, debe agregar el Tenant ID como header adicional en la configuración de Buk:

  1. En la sección Headers adicionales del webhook, agregar:
Nombre del HeaderValor
X-Tenant-Id El GUID de su tenant en Limen.
Ejemplo: 3fa85f64-5717-4562-b3fc-2c963f66afa6
Puede obtenerlo en Limen: Administración → Tenant → Información General → ID

4.5 Suscribirse a los eventos relevantes

Activar la suscripción a los siguientes tipos de eventos en Buk:

Evento BukAcción en LimenPrioridad
work_ended
worked_ended
work_finished
employee_terminated 		Bloqueo inmediato de acceso físico en hardware (todos son alias del mismo evento de baja) CRÍTICO
employee_created / work_started Registro en AuditLog — el alta de acceso la gestiona el job de sincronización periódica Normal
employee_updated / work_updated Registro en AuditLog — los cambios de datos los aplica el job de sincronización periódica Normal
Cualquier otro evento desconocido Registro en AuditLog como BUK_EVENT_UNKNOWN — Limen responde 200 para evitar reintentos de Buk Informativo
¿Por qué hay múltiples alias para la baja? Buk ha cambiado el nombre del evento de finiquito entre versiones de su API. Limen maneja todos los alias conocidos para garantizar compatibilidad independiente de la versión de Buk que use el cliente.
  1. Hacer clic en Guardar en el panel de Buk.

5. Flujo de una Baja Laboral (objetivo: < 30 segundos)

Cuando un colaborador es finiquitado en Buk, ocurre la siguiente secuencia automática:

EtapaAcción TécnicaTiempo Est.
1. Evento Buk Buk ejecuta el proceso de baja y dispara el webhook work_ended con el RUT del empleado 0s
2. Recepción Limen recibe el webhook en POST /api/webhooks/buk < 1s
3. Autenticación Verificación del Bearer Token con comparación de tiempo constante < 10ms
4. Mapeo Búsqueda del empleado por RUT en la BD de Limen ~100ms
5. Inhabilitación Marca EmployeeRecord.Status = Terminated en la BD ~50ms
6. Bloqueo Hardware Envío de comando BLOCK_USER al hardware (ZK/Hikvision) vía IAccessControlPlatformClient 2–5s
7. Auditoría Registro de SEC_REVOCATION_COMPLETE en AuditLog < 50ms

Logs esperados en Serilog para una baja exitosa

[14:30:05 INF] [BukWebhook] Tenant 3fa8...: evento 'work_ended', RUT '12345678-9'.
[14:30:05 INF] [BukWebhook] Tenant 3fa8...: iniciando flujo de baja para RUT '12345678-9'.
[14:30:05 INF] Procesando baja para tenant 3fa8..., empleado externo 12345678-9
[14:30:06 INF] Accesos revocados para empleado Juan Pérez (12345678-9).
[14:30:07 INF] [BukWebhook] Tenant 3fa8...: baja completada para RUT '12345678-9'.

6. Endpoints de la API Buk (Referencia Técnica)

Limen consulta la API de Buk para la sincronización periódica de la nómina. A continuación los endpoints reales utilizados (verificados mayo 2026):

Endpoint / Uso en LimenDescripciónParámetros clave
GET /api/v1/{country}/employees
→ GetActiveEmployeesAsync() 		Nómina activa completa (paginada). Limen usa page_size=100 y pagina automáticamente hasta agotar resultados. status=activo, page, page_size (max 100)
GET /api/v1/{country}/employees
→ GetTerminatedEmployeesAsync() 		Empleados desvinculados. Mismo endpoint, distinto filtro de status. status=inactivo, page, page_size
GET /api/v1/{country}/employees
→ GetEmployeeByRutAsync(rut) 		Búsqueda individual por RUT. Limen normaliza el RUT antes de enviarlo (sin puntos ni guión). document_number={rut_sin_puntos}, status=activo
GET /api/v1/{country}/employees
→ TestConnectionAsync() 		Prueba de conexión: trae 1 página y devuelve el total de empleados activos. status=activo&page_size=25&page=1
Nota: Limen usa siempre el mismo endpoint /employees variando el parámetro status. Buk no expone endpoints separados por tipo de empleado (no hay /employees/inactive ni /employees/terminated).

7. Verificar la Integración

7.1 Verificar conexión API (polling)

  1. Ir a Administración → Integraciones → RRHH → Buk
  2. Hacer clic en Probar Conexión
  3. El resultado debe mostrar el número de empleados activos en Buk

7.2 Verificar recepción de webhooks con Postman/curl

Simular un webhook de Buk para verificar que Limen lo procesa correctamente:

POST https://api.limen.app/api/webhooks/buk

Headers:
  Content-Type: application/json
  Authorization: Bearer {WebhookSecret configurado en Limen}
  X-Tenant-Id: {GUID del tenant}

Body:
{
  "event_type": "work_ended",
  "timestamp": "2026-05-17T14:30:00Z",
  "data": {
    "rut": "12345678-9",
    "full_name": "Juan Pérez González",
    "termination_date": "2026-05-17",
    "termination_reason": "Renuncia voluntaria"
  }
}

Respuesta esperada: HTTP 200 con {"received": true}

7.3 Verificar en AuditLog de Limen

  1. Ir a Administración → Auditoría
  2. Buscar eventos con tipo BUK_WEBHOOK_AUTH_FAILED (no deben existir si la configuración es correcta)
  3. Si el empleado existe en Limen, buscar el evento de bloqueo en hardware

8. Troubleshooting

SíntomaCausa probableSolución
"Probar Conexión" devuelve error 401 El API Key es incorrecto o expiró Regenerar el token en Buk: Configuración → API Access y actualizar en Limen
Webhook recibe HTTP 401 desde Limen El Bearer Token en Buk no coincide con el Webhook Secret en Limen Verificar que ambos valores sean idénticos (copiar/pegar, sin espacios extra)
Webhook recibe HTTP 400 "Header X-Tenant-Id requerido" No se configuró el header X-Tenant-Id en Buk Agregar el header en la configuración del webhook en Buk (ver sección 4.4)
Webhook recibe HTTP 404 "Tenant o configuración Buk no encontrada" El GUID en X-Tenant-Id no corresponde a ningún tenant con Buk configurado Verificar el Tenant ID en Limen: Administración → Tenant → ID
Baja procesada exitosamente pero el empleado no se bloquea en los torniquetes El RUT del empleado en Limen no coincide con el RUT en Buk, o el empleado no está sincronizado Ejecutar sincronización manual: Administración → Identidades → Sincronizar ahora
El job de sincronización periódica no actualiza la nómina El API Key no tiene scope de lectura de empleados Regenerar el token con scope employees:read en el panel de Buk
Webhooks de Buk llegan con event_type desconocido Buk puede cambiar nombres de eventos entre versiones Revisar logs de Limen (Serilog) para ver el event_type exacto recibido y reportar al soporte Limen
Caso crítico — Baja no procesada: Si un empleado fue finiquitado pero su acceso físico sigue activo, ejecutar el bloqueo manual inmediato desde Limen: Personas → [nombre del empleado] → Bloquear acceso. Esto actúa directamente sobre el hardware sin esperar el webhook.
